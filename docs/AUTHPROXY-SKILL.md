# AUTHPROXY-SKILL (Project Execution Standard)

## Purpose

Этот документ фиксирует рабочий стандарт внедрения дизайна и изменений в код для сайта AuthProxy.  
Цель: сохранять визуальную точность, чистую структуру и предсказуемый runtime без редизайна и случайных отклонений.

## Source Hierarchy

1. `docs/LANDING-AUTHPROXY.md` — единственный source of truth по структуре, секциям и текстам.
2. Актуальный Figma-узел задачи — source of truth по визуалу и отступам.
3. Текущая реализация в `index.html`, `styles/*`, `assets/motion/animations.js` — runtime-база.

Если есть конфликт, приоритет у `LANDING-AUTHPROXY.md` + целевого узла Figma.

## Non-Negotiable Rules

- Не менять визуальное направление без явного запроса.
- Не переписывать сайт с нуля.
- Сохранять текущий class API, токены, брейкпоинты и поведение навигации.
- Не менять контент и порядок секций вне `LANDING-AUTHPROXY.md`.
- Новые правки делать только минимально необходимыми и обратимыми.

## Implementation Protocol (Safe Batches)

1. Сначала коротко написать, что именно будет изменено.
2. Внести только один локальный батч правок.
3. Проверить, что нет поломок в соседних секциях и motion-runtime.
4. Дать краткий отчёт: что изменено, какие файлы затронуты.
5. По подтверждению — commit + push в `main`.

## Figma 1:1 Checklist

Проверка перед завершением батча:

- размеры контейнера и высоты блоков;
- внутренние отступы и интервалы между элементами;
- типографика: размер, line-height, weight, letter-spacing;
- цвета текста/фона/линий/уголков/иконок;
- выравнивание по сетке и базовым линиям;
- состояния hover/focus/active, если они есть в макете;
- соответствие ассетов (SVG/иконки) исходному узлу.

## Motion Rules (AuthProxy)

- Анимации должны оставаться спокойными, короткими и функциональными.
- Один эффект = одна цель (feedback, reveal, state).
- Не дублировать одну и ту же ответственность одновременно в CSS и JS.
- Для интенсивных эффектов учитывать `prefers-reduced-motion`.
- Любой новый motion должен быть изолирован и не ломать existing scroll/hover logic.

## Runtime QA Smoke Check

После каждого батча проверять:

- sticky header, mobile menu, dropdown/anchors;
- секции `#solution`, `#how`, `#capabilities`, `#security`, `#developers`;
- ключевые hover/focus-состояния CTA/links/buttons;
- слайдеры/счётчики/reveal-анимации после preloader.

## Communication Contract

- Коммуникация в чате — на русском.
- Формат ответа — коротко и по делу:
  - что сделал;
  - какие файлы изменил;
  - что проверить визуально.
- Для GitHub после push всегда давать:
  - ссылку на сайт;
  - ссылку на репозиторий;
  - hash коммита.
